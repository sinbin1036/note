# MCP
## 🔹 MCP란?

- **M (Model)** → AI 모델 (예: GPT, Claude, Cursor 같은 LLM)
- **C (Context)** → 모델이 이해·판단·행동할 때 필요한 맥락 (데이터, 리소스, 액션 등)
- **P (Protocol)** → AI 모델과 외부 세계를 연결하는 **약속된 규칙(조약)**

👉 정리: **MCP는 AI 모델이 외부 리소스를 안전하고 일관되게 사용할 수 있도록 정의된 표준 규칙**

---

## 🔹 왜 필요한가?

### 1. 보안 위험 방지

- MCP가 없다면 AI가 **무분별하게 명령 실행** 가능
- 예: AI가 `rm -rf /` 같은 명령을 실행 → 시스템 전체 삭제 위험
- **MCP는 권한을 제한** → "이 폴더만 읽기 가능", "DB는 SELECT만 허용"

### 2. 일관성 부족 해결

- API, DB, 파일 시스템 등마다 인터페이스가 제각각
- AI가 매번 다른 방식으로 배워야 함
- **MCP는 JSON-RPC 표준화 인터페이스 제공** → 한 방식으로 접근 가능

### 3. 신뢰성 확보

- AI가 응답 형식을 잘못 해석하면 업무 오류 발생
- **MCP는 요청/응답 구조를 강제** → 예측 가능한 상호작용 보장

### 4. 사용자 통제권 유지

- AI가 모든 걸 자동 실행하면 위험
- **MCP는 Run 버튼 같은 승인 절차**를 포함시켜 사용자가 제어 가능

---

## 🔹 MCP = 제약조건 + 안전 가이드라인

- **제약조건** → AI가 외부와 통신할 때 반드시 따라야 하는 형식/권한
- **안전 가이드라인** → 사용자가 승인·거부할 수 있고, AI가 범위 밖 행동을 못 하게 막음

---

## 🔹 비유

### 🖥️ 컴퓨터 구조의 버스(Bus)와 MCP

- **버스(Bus)**: CPU, 메모리, 입출력 장치가 데이터를 주고받는 **중앙 통로**
- **문제**: 버스에 연결된 장치들이 규칙 없이 데이터 흘리면 충돌·오작동 발생
- **해결**: 버스 프로토콜이 “누가 언제 어떻게 신호를 보낼지” 제약을 둠

➡️ **MCP도 같은 원리**

AI(=CPU)가 외부 리소스(=I/O 장치, 메모리)와 안전하게 통신하려면, **MCP라는 버스 프로토콜**이 필요함.

---

### 🔐 현실 세계에서의 비유

- **MCP 없는 세상**
    
    → AI에게 회사 컴퓨터 전체 관리자 계정 줘버린 꼴
    
    (무슨 짓을 할지 아무도 모름)
    
- **MCP 있는 세상**
    
    → "회사 문서함 중 특정 서랍만 열 수 있음. 읽기만 가능, 삭제는 불가"
    
    (필요한 일만 안전하게 수행)
    

---

## ✅ 정리

MCP는 단순히 **형식 통일**이 아니라,

AI가 **위험하지 않고, 예측 가능하며, 안전하게** 외부 자원과 상호작용하도록 만들어진 **보안·안정성 장치**다.

## 🔗 Embedding ↔ RAG의 관계

1. **문서 저장**
    
    - RAG 시스템은 외부 문서를 Vector DB(Pinecone, Weaviate, Supabase Vector 등)에 넣어둬야 함
        
    - 이때 **Embedding**을 써서 문서를 벡터화해서 저장함
        
2. **검색(Retrieval)**
    
    - 사용자가 질문: "삼성전자 매출 알려줘"
        
    - 질문도 Embedding으로 벡터화 → DB에 저장된 문서 Embedding과 비교
        
    - 가장 가까운 문서를 검색해서 LLM에 전달
        
3. **생성(Generation)**
    
    - 검색된 문서를 기반으로 LLM이 답변 작성 → 할루시네이션 줄어듦

2가지
- 학습시킬 문서를 embedding 할 워크 플로우를 짜놔야함
- 참고해서 보라고 하는 검색하는 방법을 워크 플로우를 짜놔야함

## 📌 왜 Embedding이 RAG에서 핵심인가?

1. **문서 검색의 품질 = Embedding의 품질**
    
    - 질문과 문서가 같은 의미 공간에 맵핑돼야 “관련 있는 문서”를 찾을 수 있음.
    - Embedding이 부정확하면 → 엉뚱한 문서 가져옴 → LLM이 아무리 똑똑해도 틀린 답을 냄.
        
2. **LLM이 답을 잘하는지 여부 = 검색된 컨텍스트 품질**
    
    - 좋은 Embedding = 진짜 관련 문서 찾음 → 할루시네이션 줄어듦.
    - 나쁜 Embedding = 쓸데없는 문서 붙임 → 오히려 혼란.
        
3. **확장성**
    
    - Embedding을 벡터 DB에 저장해두면 → 수십만/수백만 문서도 빠르게 검색 가능.
    - 즉, “큰 지식베이스를 LLM이 다룰 수 있게 만드는 열쇠”가 Embedding.
        

---

## ⚡ 정리

- **RAG = Retrieval + Generation**
- Retrieval의 핵심 = **좋은 Embedding을 통한 의미 기반 검색**
- 결국 **RAG 품질 = Embedding 품질**이라고 해도 과언이 아님.

# ==Product Managment (PM)==
## 정의
- 고객의 문제를 발견하고, 그 문제를 해결하는 "제품"을 만들어 회사의 "성공"까지 이끌어 내는 역할
- 무엇을 왜 만들어야 하는지 고민, 시장과 고객을 분석해 문제를 정의 하고, 제품의 성공지표(KPI)를 설정하고 이후 성과까지 책임지는 역할

---
## 업무

### 1단계 (==Why==)
==왜 이 제품을 만들어야 하는가?== (왜는 고객한테 향해 있어야 한다)
- 리서치 / 가설 수립
- 시장 조사 / 경쟁사 분석
- 제품의 비전 및 목표 설정

### 2단계 (What)
무엇을 만들어야 하는가?
- PRD, 유저 스토리 등 문서
- 디자이너/개발자와 협업, 아이디어 구체화
- **우선순위 설정**

### 3단계 (How)
개발 & 출시
- 모든팀과 끊임없는 소통 (디자인/개발/마케팅/운영 등)
- 개발과정 병목 해결/ 일정 관리
- GTM 전략 수립

#### 4단계 (성과 측정 및 개선)
우리의 가설이 맞았는가? (검증)
- 데이터분석 (사용자 행동 로그 분석, A/B 테스트 등)
- 사용자 피드백 수집/ 분석
- 이름 바탕으로 다음 버전의 제품 개선 방향 수립
- 끊임없는 개선, 반복


## 핵심역량

### ==소프트 스킬==
- 소통 : 각 포지션의 언어를 이해하고 소통하고 경청하고 설득
- 공감 : 고객의 입장에서 문제를 바라보는 능력
- 리더십 : 논리와 데이터로 동료들을 설득하고 이끄는 능력

### 하드 스킬
- 데이터 분석 : Google Analytics 등을 통한 데이터 기반 의사결정
